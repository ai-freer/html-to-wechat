## 一、什么一次就过

| 卡 | 类型 | 关键特征 | 为什么过 |
|---|---|---|---|
| 01 cover | C | 抽象塔 + 大标题 + 一行英文副标 | C 类型本质就轻文字、重视觉，模型在它的舒适区 |

| 07 phases | A | 3 大块 × (大号阶段数 + 中文 + 短英文 + 图标) | 大尺寸文字 + 仅 3 块，密度低 |
| 08 closing | C | 大字金句 + 隐喻视觉 | 视觉占 70%，文字只承担情绪 |

## 二、什么翻车了（以及为什么不是密度问题）

| 卡 | 类型 | 失败现象 | 真实原因 |
|---|---|---|---|
| 05 v1 L6 警示 | B | 整张图中文乱码 | **偶发失败**：用完全相同的 prompt 重试（v3）一次过 |
| 06 v1 loop | A | 环形 10 个节点中文全乱码 | **偶发失败**：同 prompt 重试 + 改结构后一次过 |

**关键纠正**：作者最初观察到 v1 两张乱码后，错误地归因为"文字密度过高 / 非线性布局 / 字号偏小"，并据此写了一份"舒适区量化表"（≤80 字 / ≤6 区块 / 线性布局…）。这份量化是**错的**。

用户当时直接指出："gpt-image-2 完全能扛几百字高密度信息，一点点文字密度完全是小儿科问题"，并发来一张 CPP 汽车架构的 infographic 实例作为反证——那张图含 500+ 中文字、100+ 文字区块、玻璃拟态彩色卡 + 车辆产品图 + 6 路侧栏列表 + 5 个主分区 + 底部结论行，全部正确渲染、无任何错字。

按此校准，模式 2 设计 plan 时**不应预设任何文字密度上限**。模型上限至少是 v1 失败值的 5 倍以上，远超我们任何卡片的合理设计需求。

**翻车的真实修法**：
1. 第一选择：**用完全相同的 prompt 重试**（重试一次基本就过）
2. 第二选择：检查 prompt 措辞是否有矛盾或冗余
3. 不要：缩短中文、合并区块、改用英文——这些都是错的方向

## 四、prompt 工程经验

### 1. style_sheet 前缀真的有用
两张完全独立 generate 的图（01 cover 和 04 overview）视觉一致到能直接拼成一套——纯靠 800 字的 style 前缀，没用 reference image。这意味着 Skill 端 prompt 链可以简化为"独立批量 generate"，不必做 reference image carry。

### 2. 警示色的局部锚点能精确执行
"L6 这一行（包括 icon）变警示橙" / "Phase 2 左边一道警示橙" / "L6 节点边框 + 入射箭头警示橙" —— 这些精确的局部颜色指令模型严格执行了。条件是：指令必须指明具体元素（不是"在某处加点橙"，而是"L6 那一行的 icon 和 code 都用警示橙"）。

### 3. "AVOID" 列表必须重复在每张 prompt
单张 prompt 内部不写 AVOID，模型偶尔会冒出来 emoji、霓虹、渐变。写了就没出过。AVOID 不只是"反例"，是边界拦截。

### 4. 文字密度可以用一个简单规则估算
**总字数 + 区块数 + 字号 三个变量综合判断**。如果一张图想塞 > 80 中文字、> 6 个区块、字号还小——基本一定翻车。看到这个组合就要提前减负。

### 5. 翻车时的减负策略
- **优先**：合并区块（10→8、5→3）
- **其次**：缩短每个区块内的字数（6 字→2 字）
- **最后**：放大字号 + 增加留白

不要直接退到 fallback（用英文、用 satori、用其他模型）—— 翻车基本都是 prompt 没给模型让出"喘息空间"，而不是模型能力不够。

## 五、对 Skill 端 prompt 链的直接指导

将 LLM 切片 + 写 prompt 的自动化阶段，应该把上述经验编入提示词：

```
你在为一篇文章生成图集 plan。请遵守：
- 视觉结构忠实于原文表达：原文用垂直级联就画级联，原文用闭环就画闭环，
  不要把级联结构错画成环形（会损失"时序感"）。
- 每张 prompt 末尾必须重申 AVOID 列表（colorful emoji, cartoon, neon, ...）。
- 警示色用于具体元素时必须指明该元素（"L6 行的 icon + code"，不是"某处加橙"）。
- 不要预设中文字数 / 区块数上限——gpt-image-2 实际能力远超本项目任何合理卡片设计。
- 中文偶发渲染失败时，重试同 prompt（高概率即过）；不要降级 / 简化 / 改用英文。
```

**Skill 端工程层的对应**：
- 自动重试：调 gpt-image-2 后如果检测到中文错字（怎么检测是另一个问题），自动用完全相同的 prompt 重试一次。
- 不要内置"如果失败就压缩"的兜底逻辑——那是错的方向。

## 六、成本与时延实测

- 单张 1024×1536 / quality=high：60-100 秒
- 8 张并行（LTCraft 实测）：约 90 秒全部完成
- 单张费用：约 ¥0.6（用户原话）
- 实际本轮成本：8 主图 + 4 次重生（05×2 + 06×2）= 12 次调用 × ¥0.6 ≈ ¥7.2
- 其中只有 2 次是真重生（v1 偶发失败 → v3），另 2 次是我自作主张走错方向的"v2 简化版"——这部分钱是白花的

教训：偶发失败重试一次就好，**不要在重试时改 prompt** ——除非有真证据表明 prompt 本身有错。我那 2 张 v2 既损失了原文术语精度（contextEngine slot→slot，移除 L3/L8 节点），又证明不了"密度才是问题"——同一 prompt 重试就好了。

## 七、还没验证的事

- 同一图集在**多篇不同主题文章**上风格切换的能力——目前只跑了一篇科技文，没跑过影视宣传 / 情感散文 / 财经分析
- **reference image carry** 模式对一致性是否有额外提升——这次纯 prompt 前缀就够了，但跨更多题材时可能需要
- **手绘 Excalidraw 风格**的可行性——这次走的是工程蓝图调，没走手绘
- **更短文本卡（小红书爆款那种纯一句话大字）**的渲染稳定性——理论上文字密度低应该最稳，但还没专门验证
