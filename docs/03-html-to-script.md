# 模式 3：HTML → 口播稿 Markdown

> 目标：把一篇 HTML 文章改写成可直接喂给 AI 视频生成软件（剪映 AI、Runway、可灵、Sora 类工具）的口播稿，输出 Markdown 文件。
>
> **X-C/X-A 范式归位**：本 mode 的 **X-A**（执行层）= `skill/modes/03-script/scripts/html-to-script.mjs` 纯函数（cheerio + turndown + analyze）；**X-C 通常退化为不需要** — plan 只有 4 个字段（extractMain / bannedTags / keepImages / turndownOptions），文本结构信息从 HTML DOM 直读就够。X-C 仍然有用的场景：多 article 报告 + 装饰性 chrome 干扰严重时，agent 截图后产 plan 比手填准。跨 mode 范式总论见 `skill/SKILL.md`。

## 1. 输入与输出

**输入**：HTML 内容（同模式 1，**三种来源任一**：本地文件 / 公开 URL / HTML 字符串）。

**最终交付物**：一份 `.md` 文件，符合"AI 视频生成软件可用"的口播稿格式。

**典型口播稿结构**：
```markdown
# 视频标题

## 段落 1（场景/镜头描述可选）
口语化的旁白文字，一句话一行，便于切分镜头。
不用书面语长句，避免嵌套从句。

## 段落 2
...
```

不同视频生成软件对格式要求略有差异，先按"通用 Markdown + 段落分镜"输出，覆盖大多数工具。

## 2. 核心管线

模式 3 是三个模式里最简单的，几乎全部由 LLM 完成。脚本工作量极少。

```
input.html
   │
   ├─ ① 文本提取（脚本）
   │     去掉 <style> <script> SVG 等非文本，保留段落结构
   │     输出：plain.md（基础 markdown）
   │
   ├─ ② 口语化改写（agent 直接做）
   │     LLM 按口播稿规范改写：
   │       - 书面语 → 口语
   │       - 长句 → 短句
   │       - 数据/术语 → 加解释或类比
   │       - 加自然停顿（句末/段末）
   │     输出：spoken.md
   │
   ├─ ③ 切段与镜头标记（agent 直接做，可选）
   │     按视频节奏切分（30-60 秒/段是常见单位）
   │     可选加 <!-- scene: ... --> 注释作为画面提示
   │     输出：final.md
   │
   └─ ④ 前端预览与下载
         GitHub Pages /web/script/ 页
         payload 通过 URL fragment 传入
         显示：渲染预览 / 字数 / 估算时长 / 切段标记
         按钮：复制纯文本 / 下载 .md
```

## 3. 改写规范

给 LLM 的口语化改写指令需要明确这些规则：

### 句式
- 单句 ≤ 30 字（普通话约 6-8 秒/30 字）
- 不用三层以上嵌套从句
- 主谓宾顺序，避免被动语态
- 一段一个核心意思

### 用词
- 书面词替换：「然而」→「不过」、「因此」→「所以」、「目前」→「现在」、「亦」→「也」
- 删除虚词：「的」「之」适度精简
- 数据加上下文：「47%」→「将近一半」「47%，差不多五个里有两个」
- 术语首次出现要解释：「LLM」→「大语言模型，就是 ChatGPT 那种」

### 节奏
- 句末用句号而不是逗号串联（便于 TTS 断句）
- 段末加换行作为停顿
- 重要数据/结论前加"接下来重点"类的引导词
- 反问/设问适度使用，提升节奏感

### 删除
- 表格、流程图、复杂对比 → 改写为逐项口述或干脆删除
- 代码块 → 删除或一句话概述
- 图片 alt → 删除
- 引用链接 → 删除（视频里念不出来）

## 4. 前端页面 `/web/script/`

页面非常薄，主要是预览 + 下载。技术栈：纯静态 HTML + JS，无依赖。

**功能**：
- 接收 markdown：URL fragment（`#payload=base64...`）或粘贴框
- 渲染预览（用 marked.js 或类似的轻量库）
- 显示元信息：
  - 字数
  - 估算时长（按 250 字/分钟）
  - 段落数 / 句子数
  - 平均句长
- 切段视图（按 `##` 拆分，显示每段时长）
- 下载按钮：
  - `.md` 原文
  - `.txt` 纯文本（剥离 markdown）
  - 复制到剪贴板

**两种入口**：
- 装了 Skill：agent 跑完后给一个 URL，用户点开看效果，下载
- 不装 Skill：用户粘贴自己写的 markdown 进去，看字数/时长，下载——价值有限，但保持一致的入口体验

## 5. 已知坑与设计取舍

1. **不同视频软件的格式偏好**：剪映 AI 喜欢纯文本+换行；Runway 喜欢分镜描述；Sora 类喜欢场景化叙述。先按 markdown 中性格式输出，后续按需加平台 preset。
2. **TTS 友好 vs 阅读友好**：口播稿是给 TTS 念的，不是给人读的。要避免破折号、括号说明、冒号引出（TTS 念不好停顿）。
3. **数字念法**：`2026` 是"二零二六"还是"两千零二十六"？年份/编号/数量得让 LLM 区分。
4. **专有名词读音**：英文缩写（API、URL、HTML）TTS 通常念字母没问题，但产品名（GitHub、Vercel）有时翻车。可在改写时加注音：「GitHub（嘎特哈伯）」，但需要 TTS 支持注音格式。
5. **段落长度与镜头节奏**：每段过长 → 视频画面单调；过短 → 切镜头频繁。建议每段 100-200 字（约 25-50 秒），既适合一个镜头也不会太短。
6. **HTML 里的视觉要素丢失成本**：模式 3 是三种模式里"信息丢失"最严重的——表格、图、对比都得用文字硬讲。某些重视觉的文章不适合走模式 3，应建议走模式 2。

## 6. Skill 与前端的边界

| 步骤 | 谁来做 | 在哪跑 |
|---|---|---|
| ① 文本提取 | 脚本（cheerio + turndown） | Skill 本地 Node |
| ② 口语化改写 | LLM | agent 进程内 |
| ③ 切段与镜头标记 | LLM | agent 进程内 |
| ④ 预览/下载 | 静态前端 | 浏览器（GitHub Pages） |

模式 3 的 Skill 端逻辑非常薄，主要是"提供改写规范的 prompt 模板"和"提取文本的辅助脚本"。SKILL.md 里要写清楚改写规范（即第 3 节内容），让 agent 按规范执行。

## 7. 实现路径

```
v0.1 前端 /web/script/ 占位页：
  接收 markdown，预览 + 字数 + 时长 + 下载
  没有 Skill 也能跑（粘贴模式）

v0.2 Skill 文本提取脚本：
  /skill/modes/03-script/scripts/extract-text.mjs
    HTML → 干净 markdown（纯文字）

v0.3 SKILL.md 改写规范：
  把第 3 节的所有规则写成 prompt 模板
  agent 调用时按规范改写，输出 final.md

v0.4 端到端串联：
  agent 跑完后，把 final.md base64 编进 URL fragment
  跳转到 /web/script/#payload=...
  用户点开预览 → 下载

v0.5 平台 preset（可选）：
  剪映 AI / Runway / Sora 等不同输出格式预设
  在前端加切换按钮
```

## 8. 不在范围内

- 直接生成视频（这是下游软件的事）
- TTS 合成（同上，下游软件做）
- 字幕时间轴（视频软件自动对齐 TTS 即可）
- 多语言版本（先做中文，英文/双语属于扩展）
